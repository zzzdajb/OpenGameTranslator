## 3. 技术和风格约束

技术栈：

- TypeScript + Node.js。
- CLI 工具。
- strict TypeScript，并尽量开启更严格的类型检查。
- 不做 GUI。
- 运行时插件可以是普通 JavaScript。

代码风格：

- 偏 C# / .NET 风格的 TypeScript。
- 优先使用 `class`、`interface`、`readonly`、`public`、`private`。
- 显式返回类型。
- 少用复杂箭头函数和匿名函数链式调用。
- 避免 `reduce`。
- 复杂逻辑拆成具名方法。
- 可以多用 `for...of`。
- 不做复杂 DI 容器。
- 不做抽象工厂套抽象工厂。
- 不滥用 `any`；未知 JSON 优先用 `unknown`，再做类型判断。
- 代码里面需要有适量注释，帮助新手理解关键思路；不要逐行解释，也不要堆太多注释。

代码需要新手阅读友好。

## 4. 调试纪律

**每次 Debug 迭代都伴随着极大的代价**——人工测试占用用户时间、反复构建消耗上下文、错误方向积累挫败感。因此必须选择效率最高、次数最少的排错方式。

**严禁穷举式排错。** 参考 Maya 触摸场景闪退 debug 过程：前 7 轮在无效方向上穷举（inline hook → IAT hook → 简化 detour → 逐个删 hook），最终原因是 `TextGui::getString` 的 CALL-based detour 栈偏移差了 4 字节——读一遍 detour 代码就能发现的静态 bug。

**规则：**

1. **先思考，后动手** — 面对 bug，第一步不是改代码试，而是列出所有可能性并按先验概率排序。
2. **静态分析优先** — 崩溃日志 + 代码审查能定位的问题，禁止用"改了试试"替代。
3. **请求第二意见** — 在不熟悉的领域（如 inline assembly、调用约定），必须先让 subagent 审阅。
4. **严格限制测试轮次** — debug 阶段的人工测试硬上限为 3 轮。超过 3 轮还没定位，必须停下来重新推理。
5. **每次改动必须有可证伪的假设** — "试试这样行不行"不是假设；"如果 X 是根因，则改动 Y 应该让症状 Z 消失"才是。
