## 调试纪律（Maya 触摸场景闪退教训）

**每次 Debug 迭代都伴随着极大的代价**——人工测试占用用户时间、反复构建消耗上下文、错误方向积累挫败感。因此必须选择效率最高、次数最少的排错方式。

**严禁穷举式排错。** 本次排错从 inline hook → IAT hook → 简化 detour → 逐个删 hook，前 7 轮全是在无效方向上穷举。最终原因是 `TextGui::getString` 的 CALL-based detour 栈偏移差了 4 字节——这是读一遍 detour 代码就能发现的静态 bug，不需要任何一次运行测试。

**规则：**

1. **先思考，后动手** — 面对 bug，第一步不是改代码试，而是列出所有可能性并按先验概率排序。必须先确认最可能的根因，再行动。
2. **静态分析优先** — 崩溃日志 + 代码审查能定位的问题，禁止用"改了试试"替代。本次 Event Viewer 明确报了 `c0000005` + `FaultingOffset`，日志明确写了最后加载的文件，完全足够推理。
3. **请求第二意见** — 在不熟悉的领域（如 inline assembly、调用约定），必须先让 subagent 审阅 detour 代码的正确性。一个人容易陷入思维定势。
4. **严格限制测试轮次** — debug 阶段的人工测试硬上限为 3 轮。超过 3 轮还没定位，说明方向错了，必须停下来重新推理，而不是继续穷举。每多一轮测试都是对用户时间的浪费。
5. **每次改动必须有可证伪的假设** — "试试这样行不行"不是假设。"如果 X 是根因，那么改动 Y 应该让症状 Z 消失/变化"才是。
